# Aspose.Words के साथ DOCX फ़ाइलों को पुनर्प्राप्त करने का तरीका – रिकवरी मोड सक्षम करें

क्या आपने कभी सोचा है **docx को कैसे पुनर्प्राप्त किया जाए** जब फ़ाइल खोलने से इनकार कर देती है? शायद आपको क्लाइंट‑जनित रिपोर्ट मिली है जो आपके व्यूअर को क्रैश कर देती है, या शायद नेटवर्क गड़बड़ी ने वर्ड डॉक्यूमेंट को आधा‑लिखा छोड़ दिया। ऐसे क्षणों में आप मैन्युअली पेज़ फिर से बनाने की कोशिश नहीं करना चाहते—एक बेहतर तरीका है।

अच्छी खबर यह है कि Aspose.Words for Java में एक बिल्ट‑इन **रिकवरी मोड** है जो टूटे हुए हिस्सों को खोज कर एक उपयोगी दस्तावेज़ बनाता है। इस ट्यूटोरियल में हम **रिकवरी मोड को कैसे सक्षम करें**, संभावित रूप से भ्रष्ट DOCX को लोड करें, **जाँचें कि दस्तावेज़ पुनर्प्राप्त हुआ या नहीं**, और अंत में एक साफ़ कॉपी सहेजें, यह सब दिखाएंगे। अंत तक आपके पास एक तैयार‑चलाने योग्य जावा प्रोग्राम होगा जो टूटे हुए .docx को एक नया .docx में बदल देगा—कोई मैन्युअल कॉपी‑पेस्टिंग नहीं।

> **आपको क्या मिलेगा:** एक पूर्ण, चलाने योग्य उदाहरण, प्रत्येक लाइन क्यों महत्वपूर्ण है इसका स्पष्टीकरण, एज केस के लिए टिप्स, और यह जल्दी से सत्यापित करने का तरीका कि फ़ाइल वास्तव में पुनर्प्राप्त हुई है।

---

## आवश्यकताएँ

शुरू करने से पहले सुनिश्चित करें कि आपके पास है:

- **Java Development Kit (JDK) 8+** – कोड मानक जावा APIs का उपयोग करता है।
- **Aspose.Words for Java** JAR (मार्च 2026 तक का नवीनतम संस्करण)। आप इसे Maven Central रिपॉज़िटरी से प्राप्त कर सकते हैं:

```xml
<dependency>
    <groupId>com.aspose</groupId>
    <artifactId>aspose-words</artifactId>
    <version>24.9</version>
</dependency>
```

- एक **इनपुट DOCX** जिसे आप संदेह करते हैं कि वह भ्रष्ट है (डेमो के लिए इसे `input-corrupt.docx` कहेंगे)।
- एक फ़ोल्डर जहाँ आपके पास लिखने की अनुमति है, ताकि पुनर्प्राप्त आउटपुट सहेजा जा सके।

यदि आप Maven या Gradle जैसे बिल्ड टूल का उपयोग कर रहे हैं, तो सिर्फ डिपेंडेंसी जोड़ें और आप तैयार हैं।

---

## DOCX को पुनर्प्राप्त करने का तरीका – रिकवरी मोड सक्षम करना

सबसे पहले आपको Aspose.Words को यह बताना होगा कि आपको समस्या की उम्मीद है। यह `LoadOptions` ऑब्जेक्ट को कॉन्फ़िगर करके और **रिकवरी मोड** को चालू करके किया जाता है।

```java
// Step 1: Create LoadOptions and enable recovery mode
LoadOptions loadOptions = new LoadOptions();
loadOptions.setRecoveryMode(LoadOptions.RecoveryModeEnum.RECOVER);
```

> **यह क्यों महत्वपूर्ण है:** डिफ़ॉल्ट रूप से Aspose.Words किसी भी खराब भाग को मिलने पर अपवाद फेंकेगा। `RecoveryModeEnum.RECOVER` सेट करने से लाइब्रेरी जारी रहती है और जितना संभव हो सके बचाने की कोशिश करती है। इसे एक सुरक्षा जाल की तरह समझें जो टूटे हुए हिस्सों को पकड़ लेता है, बजाय पूरे लोड ऑपरेशन को क्रैश करने के।

### प्रो टिप
यदि आप केवल समस्याओं को *लॉग* करना चाहते हैं बिना वास्तव में उन्हें ठीक किए, तो `RECOVER_WITH_WARNINGS` का उपयोग करें। `RECOVER` विकल्प वही है जिसकी आपको तब आवश्यकता होती है जब आप वास्तव में एक उपयोगी दस्तावेज़ वापस चाहते हैं।

---

## चरण 2: संभावित रूप से भ्रष्ट DOCX लोड करें

अब जब रिकवरी मोड सक्षम हो गया है, फ़ाइल को लोड करें। कंस्ट्रक्टर फ़ाइल पाथ और हमने अभी तैयार किया `LoadOptions` लेता है।

```java
// Step 2: Load the DOCX using the recovery options
String inputPath = "YOUR_DIRECTORY/input-corrupt.docx";
Document document = new Document(inputPath, loadOptions);
```

> **अंदर क्या हो रहा है?** Aspose OPC (Open Packaging Conventions) संरचना को पार्स करता है, लापता रिलेशनशिप्स को ठीक करता है, और किसी भी टूटे हुए XML फ्रैगमेंट को पुनर्निर्मित करता है। यदि फ़ाइल केवल थोड़ा क्षतिग्रस्त है, तो आपको एक पूरी तरह कार्यशील `Document` ऑब्जेक्ट मिलेगा।

### एज केस
यदि फ़ाइल *गंभीर* रूप से भ्रष्ट है (जैसे `[Content_Types].xml` भाग गायब है), तो भी Aspose एक दस्तावेज़ लौटाएगा लेकिन कई तत्व गायब हो सकते हैं। ऐसे मामलों में आप `OriginalFileInfo` को निरीक्षण करके अधिक विवरण प्राप्त कर सकते हैं।

---

## चरण 3: जाँचें कि दस्तावेज़ पुनर्प्राप्त हुआ या नहीं

लोड करने के बाद, आप लाइब्रेरी से पूछ सकते हैं कि क्या उसने कोई रिकवरी कार्य किया है। यही वह जगह है जहाँ **check document recovered** कीवर्ड काम आती है।

```java
// Step 3: Check if recovery actually occurred
boolean recovered = document.getOriginalFileInfo().isRecovered();
System.out.println("Recovered? " + recovered);
```

सामान्य कंसोल आउटपुट:

```
Recovered? true
```

यदि आउटपुट `false` है, तो फ़ाइल पहले से ही स्वस्थ थी या लाइब्रेरी उसे पुनर्प्राप्त नहीं कर सकी। आप `getOriginalFileInfo().getRecoveryWarnings()` को भी क्वेरी कर सकते हैं ताकि उन चेतावनियों की सूची मिल सके जो बताती हैं क्या ठीक किया गया।

### क्यों जाँचें
भले ही दस्तावेज़ लोड हो जाए, सूक्ष्म डेटा हानि हो सकती है (जैसे लापता इमेजेज)। पुनर्प्राप्त फ़्लैग और चेतावनियों की जाँच करके आप तय कर सकते हैं कि परिणाम स्वीकार करें या उपयोगकर्ता से अलग स्रोत माँगें।

---

## चरण 4: पुनर्प्राप्त दस्तावेज़ सहेजें

मान लीजिए रिकवरी सफल रही—या आप चेतावनियों से ठीक हैं—तो साफ़ दस्तावेज़ को लिखें। यह एक बिल्कुल नया DOCX बनाता है जिसे Microsoft Word, Google Docs, या किसी भी अन्य व्यूअर में खोला जा सकता है।

```java
// Step 4: Persist the repaired document
String outputPath = "YOUR_DIRECTORY/recovered.docx";
document.save(outputPath);
System.out.println("Recovered document saved to: " + outputPath);
```

अब आपके पास `recovered.docx` मूल टूटे हुए फ़ाइल के साथ साइड‑बाय‑साइड है। इसे Word में खोलें; आपको सभी मूल टेक्स्ट, टेबल्स, और अधिकांश इमेजेज़ intact दिखेंगे।

---

## पूर्ण कार्यशील उदाहरण

नीचे वह पूरा जावा क्लास है जो सब कुछ जोड़ता है। इसे अपने IDE में कॉपी‑पेस्ट करें, पाथ्स को समायोजित करें, और चलाएँ।

```java
import com.aspose.words.*;

public class RecoveryModeDemo {
    public static void main(String[] args) throws Exception {
        // ----------------------------------------------------
        // 1️⃣ Prepare LoadOptions to enable recovery mode
        // ----------------------------------------------------
        LoadOptions loadOptions = new LoadOptions();
        loadOptions.setRecoveryMode(LoadOptions.RecoveryModeEnum.RECOVER);

        // ----------------------------------------------------
        // 2️⃣ Load the potentially corrupted DOCX using the options
        // ----------------------------------------------------
        String inputPath = "YOUR_DIRECTORY/input-corrupt.docx";
        Document document = new Document(inputPath, loadOptions);

        // ----------------------------------------------------
        // 3️⃣ Verify whether the document was recovered
        // ----------------------------------------------------
        boolean recovered = document.getOriginalFileInfo().isRecovered();
        System.out.println("Recovered? " + recovered);

        // Optional: print any warnings (helps with debugging)
        for (String warning : document.getOriginalFileInfo().getRecoveryWarnings()) {
            System.out.println("Warning: " + warning);
        }

        // ----------------------------------------------------
        // 4️⃣ Save the recovered document
        // ----------------------------------------------------
        String outputPath = "YOUR_DIRECTORY/recovered.docx";
        document.save(outputPath);
        System.out.println("Recovered document saved to: " + outputPath);
    }
}
```

**अपेक्षित परिणाम:** जब आप प्रोग्राम चलाते हैं, कंसोल `Recovered? true` (या `false` यदि कोई रिकवरी आवश्यक नहीं थी) प्रिंट करेगा और फ़ाइल सहेजने की पुष्टि करेगा। `recovered.docx` खोलने पर एक पूरी तरह पढ़ने योग्य दस्तावेज़ दिखना चाहिए।

## सामान्य प्रश्न एवं समस्याएँ

| प्रश्न | उत्तर |
|----------|--------|
| **क्या मुझे Aspose.Words के लिए लाइसेंस चाहिए?** | हाँ, लाइब्रेरी को प्रोडक्शन उपयोग के लिए वैध लाइसेंस की आवश्यकता होती है। मूल्यांकन के लिए आप बिना लाइसेंस के कोड चला सकते हैं, लेकिन एक वॉटरमार्क दिखाई देगा। |
| **यदि फ़ाइल .doc (बाइनरी) है, .docx के बजाय?** | रिकवरी मोड दोनों फ़ॉर्मैट्स के साथ काम करता है। सिर्फ फ़ाइल एक्सटेंशन बदल दें; Aspose फ़ॉर्मैट को ऑटो‑डिटेक्ट करेगा। |
| **क्या मैं केवल विशिष्ट भाग (जैसे सिर्फ टेक्स्ट) ही पुनर्प्राप्त कर सकता हूँ?** | आप लोड करने के बाद `document.getSections()` के माध्यम से इटररेट करके जो चाहिए वह निकाल सकते हैं। रिकवरी प्रक्रिया स्वयं हमेशा पूरे पैकेज को पुनर्प्राप्त करने की कोशिश करती है। |
| **क्या रिकवरी मोड थ्रेड‑सेफ़ है?** | हाँ, प्रत्येक `Document` इंस्टेंस स्वतंत्र है। केवल `LoadOptions` को बिना उचित सिंक्रोनाइज़ेशन के कई थ्रेड्स में साझा करने से बचें। |
| **बड़ी फ़ाइलें (>100 MB) कैसे संभालें?** | `LoadOptions.setLoadFormat(LoadFormat.DOCX)` का उपयोग करके पार्सर को फोर्स करें, और JVM हीप (`-Xmx2g`) बढ़ाएँ। रिकवरी मोड थोड़ा ओवरहेड जोड़ता है लेकिन फ़ाइल आकार के अनुपात में रैखिक रहता है। |

## वास्तविक परिदृश्यों के लिए प्रो टिप्स

- **बैच प्रोसेसिंग:** डेमो कोड को एक लूप में रखें जो किसी फ़ोल्डर में `*.docx` फ़ाइलों को स्कैन करे। प्रत्येक फ़ाइल की `isRecovered` स्थिति को ऑडिट के लिए CSV में लॉग करें।
- **चेतावनियों को लॉग करना:** `getRecoveryWarnings()` सूची को एक लॉग फ़ाइल में लिखें। इससे आपको पैटर्न पहचानने में मदद मिलती है—शायद कोई थर्ड‑पार्टी ऐड‑इन दस्तावेज़ों को भ्रष्ट कर रहा है।
- **पुनर्प्राप्ति के बाद वैलिडेशन:** सहेजने के बाद, आप नया फ़ाइल फिर से लोड करके एक त्वरित sanity check चला सकते हैं (जैसे पेज काउंट की अपेक्षा से मेल खाता है या नहीं)। यह डबल‑चेक दुर्लभ एज केस पकड़ता है जहाँ पहली लोड सफल हुई लेकिन सहेजी गई फ़ाइल में अभी भी छिपी समस्याएँ हों।
- **OCR के साथ संयोजन:** यदि भ्रष्ट DOCX में स्कैन की गई इमेजेज़ हैं, तो आप पुनर्प्राप्त दस्तावेज़ को OCR लाइब्रेरी (जैसे Tesseract) में फीड करके सर्चेबल टेक्स्ट निकाल सकते हैं।

## निष्कर्ष

हमने **docx फ़ाइलों को पुनर्प्राप्त करने** के लिए Aspose.Words के रिकवरी मोड को सक्षम करने, एक टूटे हुए दस्तावेज़ को लोड करने, **दस्तावेज़ पुनर्प्राप्त हुआ या नहीं** की जाँच करने, और अंत में एक साफ़ कॉपी सहेजने का तरीका कवर किया। यह दृष्टिकोण सीधा है, केवल कुछ ही जावा लाइनों की आवश्यकता है, और अधिकांश वास्तविक‑विश्व भ्रष्टाचार परिदृश्यों में काम करता है।

अब आप जानते हैं **रिकवरी मोड को कैसे सक्षम करें**, आप इस लॉजिक को किसी भी दस्तावेज़‑प्रोसेसिंग पाइपलाइन में इंटीग्रेट कर सकते हैं—चाहे वह स्वचालित ईमेल अटैचमेंट स्कैनर हो, बैच माइग्रेशन टूल, या उपयोगकर्ता‑समक्ष अपलोड सेवा। अगले कदमों में आप `RecoveryWarning` विवरणों का अन्वेषण कर सकते हैं, या डेमो को PDFs और अन्य Office फ़ॉर्मैट्स को संभालने के लिए विस्तारित कर सकते हैं।

और प्रश्न हैं? टिप्पणी करें, कोड के साथ प्रयोग करें, और शुभ पुनर्प्राप्ति!
